# DeepSeek-v4-pro 实测表现总结

本文记录 DeepSeek-v4-pro 在代码代理与仓库理解场景下的实测表现，结论偏负面，主要问题集中在思维链长度、响应速度、编辑稳定性与幻觉控制。

## 总体结论

- 思维链过长。相较于 GLM-5，DeepSeek-v4-pro 的思维链明显更长，经常在进入实际执行前输出大量中间推理，导致交互冗长。
- 输出速度过慢。实测输出速度大约只有 30 token/s，在思维链已经偏长的前提下，整体等待时间被进一步放大，主观体验较差。
- 编辑操作稳定性不足。模型经常无法精确匹配原始代码内容，导致编辑失败或补丁无法正确落到目标位置。该问题表现得较为频繁，怀疑与其压缩 Attention 的新架构有关，但这一点目前只是现象层面的推测。

## 主要问题

### 1. 思维链过长

在需要快速试错、频繁修改代码的 agent 场景中，过长的思维链会直接拉低吞吐。与 GLM-5 相比，DeepSeek-v4-pro 更容易把大量 token 消耗在中间分析而非实际动作上。

### 2. 速度慢，且与长思维链叠加

单看 30 token/s 已经不算快；当模型同时输出很长的思维链时，用户需要等待的并不是“一个慢回复”，而是“一个很长且很慢的回复”，体验会明显恶化。

### 3. 编辑操作经常失败

在代码编辑任务中，DeepSeek-v4-pro 经常无法精确对齐原始代码片段，表现为：

- 无法正确匹配待修改的上下文
- 补丁命中失败
- 明明给了明确参照仓库，仍然改错位置或无法完成编辑

这类问题会显著影响 agent 在真实代码库中的可用性。

## 幻觉案例

### 案例 1：虚构不存在的内置 skill

任务是让 DeepSeek-v4-pro 为 `claude-code-python` 实现 skill 功能，并且已经明确要求它参考 `claude-code` 原始代码库。但模型仍然编造出 5 个完全不存在的内置 skill，例如 `commit`、`pdf`、`fix` 等，说明其在“有明确参照物”的情况下依然会凭空补全系统设定。

### 案例 2：错误判断 `.claude` 下的 skill 文件状态

在另一次实测中，要求 DeepSeek-v4-pro 检查 `.claude` 目录下是否存在 skill 文件。模型给出的结论是 `skills` 是空目录，但实际情况并非如此，目录中存在两个 skill。这个案例说明它在文件系统检查这种本应非常确定的问题上，仍然会给出与事实不符的结论。

## 结论

如果使用场景是代码代理、仓库改造或需要高可靠文件编辑的工程任务，DeepSeek-v4-pro 当前暴露出的几个问题会叠加放大：

- 长思维链拉高 token 消耗
- 低输出速度拉长等待时间
- 编辑失败降低执行成功率
- 幻觉进一步破坏结果可信度

综合来看，DeepSeek-v4-pro 在上述场景中的实用性和体验均不理想。
